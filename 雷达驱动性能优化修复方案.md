## 🔧 修复步骤

### 修复 1：优化点云内存释放（安全，强烈推荐）

**文件**: `src/rslidar_sdk/src/rs_driver/src/rs_driver/driver/lidar_driver_impl.hpp`

**修改位置**: 第 192-203 行的 `getPointCloud()` 函数

```cpp
// 修改前（有问题）：
template <typename T_PointCloud>
std::shared_ptr<T_PointCloud> LidarDriverImpl<T_PointCloud>::getPointCloud()
{
  while (1)
  {
    std::shared_ptr<T_PointCloud> cloud = cb_get_cloud_();
    if (cloud)
    {
      cloud->points.resize(0);  // ❌ 不会释放内存
      return cloud;
    }

    LIMIT_CALL(runExceptionCallback(Error(ERRCODE_POINTCLOUDNULL)), 1);
  }
}

// 修改后（修复）：
template <typename T_PointCloud>
std::shared_ptr<T_PointCloud> LidarDriverImpl<T_PointCloud>::getPointCloud()
{
  while (1)
  {
    std::shared_ptr<T_PointCloud> cloud = cb_get_cloud_();
    if (cloud)
    {
      // ✅ 使用 swap 技巧真正释放内存
      std::vector<typename T_PointCloud::PointT> temp;
      cloud->points.swap(temp);
      return cloud;
    }

    LIMIT_CALL(runExceptionCallback(Error(ERRCODE_POINTCLOUDNULL)), 1);
  }
}
```

**影响评估**:
- ✅ **正面**: 解决内存持续增长问题
- ✅ **无副作用**: swap 是标准 C++ 做法
- ✅ **性能**: 几乎没有性能损失

### 修复 2：优化 Socket 接收缓冲区（保守方案）

**文件**: `src/rslidar_sdk/src/rs_driver/src/rs_driver/driver/input/unix/input_sock_select.hpp`

**修改位置**: `createSocket()` 函数中的 `#ifdef ENABLE_MODIFY_RECVBUF` 部分

```cpp
// 修改前：
#ifdef ENABLE_MODIFY_RECVBUF
  {
    uint32_t opt_val = input_param_.socket_recv_buf;  // 可能未初始化
    // ...
    if (opt_val < 1024)
    {
      opt_val = 106496;  // 104 KB，太小
    }
    // ...
  }
#endif

// 修改后：
#ifdef ENABLE_MODIFY_RECVBUF
  {
    uint32_t opt_val = input_param_.socket_recv_buf;
    uint32_t before_set_val = 0;
    uint32_t after_set_val = 0;
    // ✅ 设置合理的默认值（512 KB）
    if (opt_val < 1024 || opt_val > 1048576)  // 限制在 1MB 以内
    {
      opt_val = 524288;  // 512 KB，保守但足够
    }
    socklen_t opt_len = sizeof(uint32_t);
    
    // get original value
    if (getsockopt(fd, SOL_SOCKET, SO_RCVBUF, &before_set_val, &opt_len) == -1)
    {
      perror("getsockopt before");
      return -1;
    }
    RS_INFO << "Original receive buffer size: " << before_set_val << " bytes" << RS_REND;

    // set new value
    if (setsockopt(fd, SOL_SOCKET, SO_RCVBUF, &opt_val, opt_len) == -1)
    {
      perror("setsockopt");
      return -1;
    }

    // get new value
    if (getsockopt(fd, SOL_SOCKET, SO_RCVBUF, &after_set_val, &opt_len) == -1)
    {
      perror("getsockopt after");
      return -1;
    }
    RS_INFO << "After setting: receive buffer size: " << after_set_val << " bytes" << RS_REND;
  }
#endif
```

**影响评估**:
- ✅ **正面**: 减少网络丢包
- 🟡 **注意**: 内存占用增加约 512KB/socket
- ⚠️ **保守**: 设置为 512KB 而非 2MB，更安全

### 修复 3：QoS 深度调整（⚠️ 高风险，暂不推荐）

**为什么暂时不改？**

1. **Fast-LIO 需要最新数据**：深度过大会导致 Fast-LIO 处理旧数据
2. **可能引入延迟**：队列越深，消费者可能拿到更旧的帧
3. **需要全局评估**：所有订阅 `/airy_points` 的节点都可能受影响

**如果未来要改，建议先测试**:
```cpp
// 在 config.yaml 中先测试这个值：
ros_queue_length: 20  # 从 100 降到 20，减少延迟
```

## 🚨 可能的问题和解决方案

### 问题 1：修改后编译失败

**原因**: 可能是 C++ 标准版本问题

**解决**:
```bash
# 检查 CMakeLists.txt 中的 C++ 标准
grep -n "CXX_STANDARD" src/rslidar_sdk/CMakeLists.txt
# 应该是 C++14 或 C++17
```

### 问题 2：修改后雷达不发布数据

**原因**: Socket 缓冲区设置可能被系统限制

**解决**:
```bash
# 检查系统最大缓冲区
cat /proc/sys/net/core/rmem_max
cat /proc/sys/net/core/rmem_default

# 如果需要，临时增加（需要 sudo）
sudo sysctl -w net.core.rmem_max=1048576
sudo sysctl -w net.core.rmem_default=524288
```

### 问题 3：性能反而下降

**原因**: 可能是 swap 操作太频繁

**解决**: 回退修复 1，只保留修复 2

## 💡 其他优化建议（无需修改源码）

### 1. 调整配置文件

**文件**: `src/rslidar_sdk/config/config.yaml`

```yaml
driver:
  use_lidar_clock: true      # 使用雷达时钟，减少时间戳计算
  dense_points: true         # 丢弃 NAN 点，减少数据量
```

### 2. 使用 epoll 代替 select（需要重新编译）

```bash
# 修改 CMakeLists.txt
option(ENABLE_EPOLL_RECEIVE "Receive packets with epoll() instead of select()" ON)

# 重新编译
colcon build --packages-select rslidar_sdk
```

如果业务允许，可以在配置中跳过某些帧：
